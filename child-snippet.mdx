The `child` snippet gives you complete control over the rendered elements in Bits UI components. It lets you substitute your own element or custom component while the Bits UI component continues to handle all accessibility attributes, event handlers, and internal state. This is the primary mechanism for render delegation in Bits UI.

## When to use it

Use the `child` snippet when you need:

- Svelte-specific features: transitions, animations, actions, or scoped styles
- Integration with a custom component from your design system
- Precise control over the DOM structure
- Advanced composition of multiple components

## Basic usage

Many Bits UI components render a default HTML element. For example, `Accordion.Trigger` renders a `<button>` by default:

```svelte
<button {...props}>
  {@render children()}
</button>
```

When you need to customize this element, the `child` snippet lets you take control:

```svelte
<script lang="ts">
  import MyCustomButton from "$lib/components";
  import { Accordion } from "bits-ui";
</script>

<!-- Using a custom component -->
<Accordion.Trigger>
  {#snippet child({ props })}
    <MyCustomButton {...props}>Toggle Item</MyCustomButton>
  {/snippet}
</Accordion.Trigger>

<!-- Using a native element with scoped styles -->
<Accordion.Trigger>
  {#snippet child({ props })}
    <button {...props} class="scoped-button">Toggle Item</button>
  {/snippet}
</Accordion.Trigger>

<style>
  .scoped-button {
    background-color: #3182ce;
    color: #fff;
  }
</style>
```

The `props` parameter contains all necessary attributes and event handlers. Spreading `{...props}` onto your element applies them correctly.

## How it works

When you use the `child` snippet:

1. The component merges its internal props with any props you passed to the component
2. The merged props are provided to your snippet via the `props` parameter
3. You decide which element or component receives those props
4. The component's internal logic continues to work correctly

Components that support `child` are implemented roughly like this:

```svelte
<script lang="ts">
  // Bits UI component internal logic
  let { child, children, ...restProps } = $props();
  const trigger = makeTrigger();

  // Merge internal props with user props
  const mergedProps = $derived(mergeProps(restProps, trigger.props));
</script>

{#if child}
  {@render child({ props: mergedProps })}
{:else}
  <button {...mergedProps}>
    {@render children?.()}
  </button>
{/if}
```

## Working with props

### Custom IDs and attributes

Pass custom IDs, event handlers, or other attributes to the component itself — they will be included in the merged `props` object:

```svelte
<Accordion.Trigger
  id="my-custom-id"
  data-testid="accordion-trigger"
  onclick={() => console.log("clicked")}
>
  {#snippet child({ props })}
    <button {...props}>Open accordion item</button>
  {/snippet}
</Accordion.Trigger>
```

The `props` object will include:

- Your custom `id="my-custom-id"`
- Your `data-testid="accordion-trigger"`
- Your click handler, properly merged with internal handlers
- All required ARIA attributes

## Combining with Svelte features

Apply Svelte-specific features like transitions, actions, and scoped styles to your custom elements:

```svelte
<Accordion.Trigger>
  {#snippet child({ props })}
    <div {...props} use:myCustomAction class="my-custom-trigger">
      <!-- ... -->
    </div>
  {/snippet}
</Accordion.Trigger>

<style>
  .my-custom-trigger {
    background-color: #3182ce;
    color: #fff;
  }
</style>
```

### With Svelte transitions

```svelte
<Dialog.Content>
  {#snippet child({ props, open })}
    {#if open}
      <div {...props} transition:scale={{ start: 0.95 }}>
        Dialog content with a scale transition
      </div>
    {/if}
  {/snippet}
</Dialog.Content>
```

## Floating components

Floating content components — tooltips, popovers, dropdowns, and similar — require a two-level structure due to their positioning requirements.

### Required structure

1. An **outer wrapper element** with `{...wrapperProps}` — handles positioning via Floating UI
2. An **inner content element** with `{...props}` — receives your styles and transitions

```svelte
<Popover.Content>
  {#snippet child({ wrapperProps, props, open })}
    {#if open}
      <div {...wrapperProps}>
        <div {...props}>
          <!-- ... -->
        </div>
      </div>
    {/if}
  {/snippet}
</Popover.Content>
```

### Floating element example with transitions

```svelte
<Tooltip.Content>
  {#snippet child({ wrapperProps, props, open })}
    {#if open}
      <div {...wrapperProps}>
        <div {...props} transition:fade>Custom tooltip content</div>
      </div>
    {/if}
  {/snippet}
</Tooltip.Content>
```

### Important rules for floating content

<Warning>
  Never style the wrapper element. Floating UI uses it for positioning calculations. All visual styling must go on the inner content element.
</Warning>

- The `open` parameter lets you conditionally render the content, triggering Svelte transitions
- Always maintain the two-level structure; omitting the wrapper will break positioning

### Components requiring wrapper elements

- `Combobox.Content`
- `DatePicker.Content`
- `DateRangePicker.Content`
- `DropdownMenu.Content`
- `LinkPreview.Content`
- `Menubar.Content`
- `Popover.Content`
- `Select.Content`
- `Tooltip.Content`

## Common pitfalls

<AccordionGroup>
  <Accordion title="Missing props spread">
    Always include `{...props}` on your custom element. Without it, the component loses all accessibility attributes and event handlers.
  </Accordion>
  <Accordion title="Styling the wrapper element">
    Never apply styles to the outer wrapper element in floating components. It is managed by Floating UI for positioning. Style the inner content element instead.
  </Accordion>
  <Accordion title="Children ignored when using child snippet">
    When the `child` snippet is used, other children passed outside the snippet are ignored. Put all content inside the snippet.
  </Accordion>
  <Accordion title="Missing two-level structure for floating elements">
    Forgetting the `wrapperProps` wrapper element on floating components will break positioning entirely. Always use both the outer wrapper and the inner content element.
  </Accordion>
</AccordionGroup>
